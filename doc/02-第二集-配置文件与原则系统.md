## 先从招聘剪辑助理说起

假设你要招聘一个**剪辑助理**来帮你做事。

**错误的招聘方式：**
> "你好，帮我剪辑视频。"

这个助理会很困惑——什么叫"剪辑"？
- 是把长视频剪短？
- 还是把多个视频拼成一个？
- 还是做特效？

**正确的招聘方式：**
> "你好，帮我处理口播视频。具体做法是：
> 1. 把'嗯、啊、那个、然后'这样的语气词删掉
> 2. 中间停顿超过2秒的地方也删掉
> 3. 如果我说错了然后纠正了，保留后面那句正确的
> 4. 重复的内容只保留最后一次说的
> 5. 有疑问的地方先标出来，我来决定"

看到了吗？**明确规则才能做好事情。**

这一步我们就是在做这件事——给我们的程序"助理"定好工作守则。

---

## 规则怎么定？

我们从**最常见的口播场景**出发，想想你自己录视频的时候哪些内容会想删掉：

### 场景1：语气词

你录视频时会不自觉地说：

> "嗯...那个...今天呢...啊...我想跟大家聊一聊..."

这些词对内容没有任何帮助，可以删。

**语气词清单（可继续补充）：**
- 嗯、啊、呃、哦、额
- 那个、这个
- 然后（作为连接词的"然后"，不是内容里的"然后"）
- em、uh、umm（英文常见）

### 场景2：停顿

你在思考下一句要说什么，视频里出现了几秒的安静：

> "我们要定目标...（沉默了3秒）...然后拆解成小步骤..."

太长的停顿会让视频拖沓，影响观看体验。

**停顿处理建议：**
- 停顿 < 0.5秒 → 正常呼吸，不用管
- 停顿 0.5-1秒 → 看情况（可以不删）
- 停顿 1-2秒 → 建议删除（你在想事情）
- 停顿 > 2秒 → 必须删除（太长了）

### 场景3：口误和纠正

你说错了，然后自己纠正：

> "今天的目标是100...哦不对，是1000元。"

这种情况：
- ❌ 删前面那句错的（"今天的目标是100..."）
- ✅ 保留后面那句正确的（"是1000元"）

### 场景4：重复表达

你为了强调或者没想好，重复说了一遍：

> "今天要讲三件事...三件事：第一...第二...第三..."

重复的内容只保留一次就行。

## 最重要的一条规则：语义完整性优先

在讲具体规则之前，先讲**最重要的一条**：

### 🎯 核心原则：**宁可少删，不要错删**

举个例子：

> "嗯...今天我想跟大家分享一个...那个...非常重要的观点。"

这里的"嗯"可以删，"那个"也可以删。
但是"非常重要的观点"——哪怕前后有停顿，也**绝对不能删**。

**因为：**
- 删掉语气词 = 视频更紧凑 ✅ 好的
- 删掉停顿 = 节奏更快 ✅ 好的

---
版权声明：[https://mowanyan.com](https://mowanyan.com)
- 删掉内容 = 信息丢失 ❌ 糟糕！

**所以，我们的剪辑原则是：**

> 如果不确定该不该删 → **不删**
> 如果删了可能影响理解 → **不删**
> 如果是内容的一部分 → **不删**

就像做菜，盐放少了可以加，但放多了就回不去了。剪辑也是一样——少删了还能再剪，剪错了就找不回来了。

## 用表格整理规则

把刚才讲的整理成一张"工作守则"：

| 类型 | 例子 | 处理方式 | 注意事项 |
|------|------|----------|---------|
| 语气词 | "嗯、啊、那个、然后" | ✅ 删除 | 只删独立的语气词，不要把句子里的"然后"都删了 |
| 短停顿 | < 0.5秒 | ❌ 不删 | 正常说话节奏 |
| 中等停顿 | 0.5-1秒 | ⚠️ 可选 | 看情况 |
| 长停顿 | > 1-2秒 | ✅ 删除 | 明显是在想下一句 |
| 超长停顿 | > 2秒 | ✅ 必须删除 | 太拖沓了 |
| 口误后纠正 | "100...哦不对，是1000" | ✅ 删前面的 | 保留后面正确的 |
| 重复内容 | "三件事...三件事..." | ✅ 删前面的 | 保留后面的 |
| 内容有疑问 | 不确定是不是废话 | ❌ 先不删 | 标出来让用户决定 |

## 用一个完整的例子理解

我们来看看一段真实的口播，然后应用上面的规则：

**原始内容：**
> "嗯...那个...大家好。啊...今天呢...我想跟大家分享一个...那个...嗯...关于学习效率的话题。首先呢...我们要...呃...先定方向...哦不对...应该先定目标。对，先定目标，然后呢...拆解成小步骤。嗯...那个...其次呢...要保持专注。"

**应用规则后：**
> "大家好。今天我想跟大家分享一个关于学习效率的话题。首先我们要先定目标，对，先定目标，然后拆解成小步骤。其次要保持专注。"

**具体哪里删了：**

| 时间 | 原始内容 | 处理 | 原因 |
|------|---------|------|------|
| 0-0.5秒 | "嗯..." | ✅ 删 | 语气词 |
| 0.5-1秒 | "那个..." | ✅ 删 | 语气词 |
| 1-2秒 | "大家好。" | ✅ 保留 | 内容 |
| 2-2.5秒 | "啊..." | ✅ 删 | 语气词 |
| 2.5-5秒 | "今天呢...我想跟大家分享一个..." | ⚠️ 部分删 | "呢"是语气词，删掉；中间的停顿看长度 |
| 5-6秒 | "那个..." | ✅ 删 | 语气词 |
| 6-7秒 | "嗯..." | ✅ 删 | 语气词 |
| ... | ... | ... | ... |
| 30-35秒 | "先定方向...哦不对...应该先定目标。" | ✅ 删前面的 | 口误后纠正，保留后面的 |

## 这些规则存在哪里？

这些规则我们需要写成文件，放在 `config/principles/` 目录下：

```
config/
├── principles/
│   ├── 1-core.md          ← 核心原则（最重要的那条）
│   ├── 2-fillers.md       ← 语气词规则
│   ├── 3-pauses.md         ← 停顿规则
│   ├── 4-corrections.md   ← 纠错规则
│   └── README.md           ← 规则说明（索引）
│
└── preferences/
    └── user.json          ← 用户的喜好设置
```

为什么要分开存？

- **核心原则（1-core.md）** - 最顶层的规则，其他规则都不能违反它
- **分类规则（2-xxx、3-xxx、4-xxx）** - 具体怎么处理各种情况
- **用户偏好（user.json）** - 比如"这个用户喜欢激进一点的剪辑风格，多删一点"

**就像法律体系：**
- 宪法（核心原则）= 最高准则
- 具体法律（分类规则）= 具体怎么判
- 司法解释（用户偏好）= 针对具体情况的调整

## 核心原则文件（1-core.md）

这个文件就写一句话，但非常重要：

```markdown
# 核心剪辑原则

## 🎯 最高准则：语义完整性优先

### 核心理念

宁可不删，不要错删。任何剪辑决策都不能破坏内容的完整性和原意。

### 删前保后原则

口播过程中，后说的内容通常更完整、更准确。

**应用场景：**
- ✅ 口误后纠正：删除口误部分，保留纠正后的
- ✅ 表达不顺畅后重说：删除不顺畅的部分，保留重说的
- ✅ 重复表达：删除前面重复的，保留最后一次表达

### 最小干预原则

只删除明确是"废话"的内容，保持原始表达风格。

**判断标准：**
- ❌ 不删除任何承载信息的词汇
- ❌ 不删除逻辑连接词（除非过度使用）
- ✅ 只删除填充性语气词和停顿

### 上下文感知原则

判断一个片段是否该删除时，必须结合上下文综合判断。

**例子：**
- "那个"作为填充词可以删除
- 但"那个问题很重要"中的"那个"是指示代词，不能删除

### 人工复核原则

AI 的判断仅供参考，最终决策由用户确认。

**实践方式：**
- 自动标记可疑内容，但不自动执行删除
- 提供审核页面让用户确认
- 保存用户选择，下次参考
```

## 语气词规则（2-fillers.md）

```markdown
# 语气词处理规则

## 常见语气词清单

### 必须删除的填充词

| 词 | 出现场景 | 例子 |
|----|---------|------|
| 嗯/恩 | 思考、停顿 | "嗯...我觉得..." |
| 啊 | 停顿、强调 | "啊...对的..." |
| 呃 | 卡壳、思考 | "呃...那个..." |
| 哦 | 回应、思考 | "哦...是这样的..." |
| 那个 | 填充、指代 | "那个...我们来看..." |
| 这个 | 填充、指代 | "这个...问题是..." |
| em/uh/umm | 英文常见填充 | "em... let's see..." |

### 需要谨慎处理的词

这些词有时候是内容的一部分，不能一概而论：

| 词 | 何时可以删 | 何时不能删 |
|----|-----------|-----------|
| 然后 | 作为连接词过度使用时 | 表示顺序/逻辑时 |
| 对 | 自问自答的填充词 | 表示确认时 |
| 好 | 过渡词 | 表示评价时 |

## 判断方法

1. **位置判断**：句子开头的语气词基本可以删
2. **语义判断**：去掉后不影响理解的可以删
3. **频率判断**：连续出现多个语气词，删除重复的

## 例子

✅ 可删：
> "嗯...啊...今天想聊的是..." → "今天想聊的是..."

❌ 不可删：
> "那个问题很重要" → "那个"是指示代词，不能删

⚠️ 谨慎：
> "然后呢...然后我们来看..." → 可以删一个"然后"，但不要全删
```

## 停顿规则（3-pauses.md）

```markdown
# 停顿处理规则

## 停顿时长分级

| 时长 | 类型 | 建议处理 | 说明 |
|------|------|---------|------|
| < 0.5秒 | 自然呼吸 | ❌ 不处理 | 正常说话的停顿 |
| 0.5-1秒 | 短暂停顿 | ⚠️ 可选 | 可能是正常节奏 |
| 1-2秒 | 较长停顿 | ✅ 建议处理 | 大概率在想下一句 |
| > 2秒 | 长停顿 | ✅ 必须处理 | 太拖沓，影响观看 |

## 特殊情况

### 语义停顿

有时候停顿是为了强调：

> "记住这一点...（停顿）...非常重要。"

这里的停顿是说话人有意的，**不应该删除**。

**判断方法：**
- 后面跟着重要内容（"记住"、"重点是"、"关键是"）
- 停顿时长虽长，但前后内容关联紧密
- 有强调的语气

### 情感停顿

表达情感时的停顿：

> "那天...（停顿）...我真的很难过。"

这种停顿表达了情绪，**建议保留**。

## 用户偏好设置

除了固定规则，我们还可以让用户自己调整。

想象你去理发店：
- 有的顾客说："剪短点"
- 有的顾客说："稍微修一下就行"

我们的工具也一样，有的用户喜欢：
- **保守模式**：少删一点，保留更多原味
- **标准模式**：按默认规则处理
- **激进模式**：能删的都删掉，追求极致紧凑

这些设置存在 `config/preferences/user.json` 里：

```json
{
  "version": "1.0.0",
  "lastUpdated": "2026-06-08",

  "settings": {
    "outputDir": "./output",
    "autoReview": true,
    "mode": "standard"
  },

  "thresholds": {
    "silence": 1.5,
    "fillerSensitivity": "medium"
  }
}
```

**各字段说明：**

| 字段 | 作用 | 可选值 |
|------|------|--------|
| `mode` | 剪辑风格 | `conservative`（保守）/ `standard`（标准）/ `aggressive`（激进） |
| `silence` | 停顿阈值（秒） | 1.0 = 停超过1秒就标记 |
| `fillerSensitivity` | 语气词敏感度 | `low`（保守）/ `medium`（适中）/ `high`（激进） |

## 小结：这一步我们做了什么？

**核心成果：**

1. ✅ 定好了核心规则（语义完整性优先）
2. ✅ 定好了具体处理规则（语气词、停顿、纠错...）
3. ✅ 设计了用户偏好系统（让用户可以调整风格）

**你可能会问的问题：**

**Q: 规则定这么死，会不会太僵化？**
A: 规则是给 AI 的参考，不是必须严格执行。最后会给用户一个审核页面，让用户自己决定。

**Q: 我想加一些自定义规则怎么办？**
A: 很简单，打开配置文件加进去就行。比如你可以加一个"口头禅规则"来处理自己的口头禅。

**Q: 规则会不会过时？**
A: 会的，所以我们把规则存在配置文件里，随时可以改，不需要重新写程序。
